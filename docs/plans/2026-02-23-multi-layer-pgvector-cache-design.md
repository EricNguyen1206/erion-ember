# Multi-Layer Semantic Cache Architecture Design (L1/L2)

## 1. Overview
This document outlines the architecture for adding a high-performance, multi-layered caching system to the Erion Ember project. 
The objective is to implement an ultra-fast L1 exact-match cache backed by an L2 semantic (vector) cache utilizing `pgvector` in PostgreSQL. This design adheres to SOLID principles, prioritizes lean/fast execution, and ensures production-grade resilience against network partitions or database outages.

## 2. Architecture & Data Flow

### The Multi-Layer Structure
*   **Layer 1 (L1) - In-Memory Exact Match:** An O(1) doubly-linked list LRU Map (reusing our existing `MetadataStore`). It uses `xxhash-addon` for 10x faster hashing of the normalized prompt. It guarantees ultra-low latency for exact text hits.
*   **Layer 2 (L2) - Durable Semantic Search:** A PostgreSQL database enhanced with the `pgvector` extension. This replaces the existing local vector indexes (`hnswlib-node` and `annoy.js`). Connections are managed via the lean, SQL-centric `postgres.js` pool.

### Data Flow (Query - Read Path)
1.  **L1 Exact Check:** The incoming prompt is hashed (`xxhash`). We query the L1 LRU Map. If found, return immediately (Cache Hit).
2.  **L2 Semantic Check:** If L1 misses, we generate the embedding vector. We query the `pgvector` database using a cosine similarity (`<=>`) threshold.
3.  **L2 Hit / Promotion:** If a semantic match is found in L2, we return the cached response and *promote* this prompt/response pair into the L1 cache for future O(1) exact matches.
4.  **Cache Miss (LLM Call):** If both miss, the prompt goes to the LLM.

### Data Flow (Write-Behind - Write Path)
When a new response is generated by the LLM:
1.  Immediately save it to the L1 in-memory cache.
2.  Return the response to the user.
3.  *Asynchronously* fire a background job to insert the prompt, embedding vector, and response into the PostgreSQL L2 cache. This ensures the user isn't blocked by database I/O.

## 3. Components & Database Schema

### Component Responsibilities (SRP)
*   **`L1Cache` (In-Memory Map):** Wraps `MetadataStore`. Handles O(1) lookups, memory bound enforcement, and TTL eviction using `xxhash`.
*   **`L2Cache` (pgvector):** Implements the `VectorIndex` interface. Manages `postgres.js` pool, inserts vectors, and executes cosine similarity searches.
*   **`SemanticCache` (Orchestrator):** Coordinates the L1 -> L2 fallback flow, invoking the `Quantizer`/`Compressor`, and dispatching asynchronous background writes to L2.

### Database Schema (PostgreSQL)
```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS semantic_cache (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    hash TEXT UNIQUE NOT NULL, -- xxhash for exact matching/deduplication
    prompt TEXT NOT NULL,
    response TEXT NOT NULL,
    embedding vector(1536) NOT NULL,
    metadata JSONB,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    expires_at TIMESTAMP WITH TIME ZONE
);

-- Crucial: HNSW index on the vector column using cosine distance
CREATE INDEX ON semantic_cache USING hnsw (embedding vector_cosine_ops);
-- Fast point lookups for Write-Behind upserts
CREATE INDEX ON semantic_cache (hash);
```

## 4. Error Handling & Testing Strategy

### Error Handling & Fallbacks
*   **Graceful Degradation:** The cache must never bring down the primary LLM pipeline. If Postgres is unreachable, `SemanticCache` catches the error, logs a warning to `stderr` (MCP standard), and gracefully degrades to L1/LLM only.
*   **Write-Behind Resilience:** The asynchronous Postgres writes will use a bounded retry queue. If a write fails repeatedly, it is dropped from L2 (but remains in L1) to prevent memory bloating from a stalled queue.

### Testing Strategy (Isolated Docker Environment)
To ensure the build size and runtime performance of the project are not impacted, testing will be strictly isolated:

1.  **Dockerized Integration:** A `docker-compose.test.yml` will spin up a lightweight PostgreSQL `pgvector` container exclusively during the test phase (`bun test`).
2.  **Dev/Prod Separation:** The `pgvector` image and `postgres.js` driver will be declared as standard dependencies (or `devDependencies` for the test infrastructure) ensuring they don't bloat the final production bundle if unused by the end-user (though `postgres.js` is extremely small).
3.  **Memory Leak Prevention:** Strict checks in `afterEach` ensure both the L1 `MetadataStore` (O(1) linked list) and the Postgres connection pool are completely destroyed, preventing dangling native handles in Bun.
